Day 11 · 工作流配置实战

BRF+规则配置步骤、审批层级实现、测试脚本编写
⏱ 约65分钟🔧 BRF+ 核心配置

11.1 BRF+配置入门：从零到第一条规则

今天你将亲历BRF+的配置步骤。以下以"物料利润中心必填"为例演示完整配置流程：

Step 1：进入BRF+ Workbench

操作路径：
事务码：BRF+（或 Transaction BRFPLUS）
→ Create Application（新建应用）
→ Application Name: HLD_MDG_RULES
→ Application Group: Master Data Governance

Step 2：创建Function（函数）

操作：在Application下 Create Function
→ Function Name: HLD_MAT_VALIDATE（物料数据校验）
→ Function Type: Condition（条件类型，返回True/False）
→ Context（上下文）: 选择MDG Material Data Model

Step 3：创建Ruleset和Rule

操作：
→ Create Ruleset: HLD_MAT_RULES
→ Create Rule: HLD_RULE_PC_MANDATORY（利润中心必填）
→ Condition: IF PROFIT_CENTER IS INITIAL（为空） THEN → Return Error Message "利润中心为必填项（E001）"
→ Activate（激活规则）

Step 4：将BRF+ Function关联到MDG CR类型

关联路径：
SPRO → MDG → General Settings → Process Modeling → Change Requests →
→ Configure Change Request Types → HLD_MATCREATE
→ Validation: 指向 HLD_MAT_VALIDATE Function

📝 知识测验

Q1. BRF+ Function的"Context（上下文）"配置的作用是什么？
✅ 解析：Context定义规则的数据范围：如果Context选择MDG Material Data Model，则该Function可以访问物料的所有字段（物料类型、利润中心等）。没有正确Context，无法引用字段进行校验。
Q2. 在BRF+规则中，RULE-M-003（物料描述长度<5字符→警告）与RULE-M-001（利润中心为空→报错）的关键区别是？
✅ 解析：MDG校验规则有两级：Error（报错，阻断提交）和Warning（警告，提示后可继续）。利润中心必填是硬性要求，用Error；描述长度只是推荐规范，用Warning提示。
Q3. 多法人动态路由通过BRF+的哪种功能实现？
✅ 解析：Decision Table是BRF+中实现多值映射的最佳方式：输入列=工厂代码，输出列=对应审批人。维护简单，业务用户也能理解，且不需要ABAP开发即可调整路由规则。
Q4. 工作流场景C（退回修改）中，申请人修改数据重新提交后，工作流从哪步重新开始？
✅ 解析：通常配置为从Revision所在步骤（或退回到的步骤）重新开始，而不是完整走全程。这避免了简单修改后还需要从头再走多步审批的效率问题。但具体可按业务需求配置。
Q5. RULE-BP-003（申请人≠审批人）防止的具体风险是什么？
✅ 解析：这是SAP MDG最重要的内控规则之一：银行账号变更如果一人发起、一人审批（同一人），则内部员工可以修改供应商银行账号到自己名下，下次付款时资金转入个人账户。双人审批彻底堵上这个漏洞。
