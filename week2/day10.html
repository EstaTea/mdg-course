10.4 Task 3：MDG-BP CR类型设计（含反欺诈）

CR类型场景工作流特殊内控
HLD_BP_CUST_NEW新建客户型BP初审（1天）→ SD Owner（2天）→ 激活信用代码唯一性校验
HLD_BP_VEND_NEW新建供应商型BP初审（1天）→ MM Owner（2天）→ 激活银行账号格式校验
HLD_BP_BANK_CHG修改供应商银行账号申请人发起 → MM Owner审批（1天）→ 财务Controller确认（1天） → 激活⚠️ 双人审批；申请人≠审批人；发邮件通知财务VP
HLD_BP_BLOCK冻结/停用BPMM Owner（供应商）或SD Owner（客户）审批→ 激活检查未结应收/应付
HLD_BP_BOTH_NEW新建双重身份BP（关联方）初审→ MM Owner + SD Owner 并行审批 → 激活关联方标识强制打标
银行账号变更防欺诈规则（关键！）：
1. CR申请人必须与审批人不同（BRF+权限规则）
2. 审批链：申请人发起 → MM Owner审批 → 财务Controller二次确认
3. 激活时自动发邮件给财务VP（通知，非审批）
4. 系统记录银行账号变更日志，不可删除（审计追踪）

10.5 Task 4：重复客户合并方案

华灯集团有850对重复客户（同一经销商在多法人各有一条），合并策略如下：

Step 1识别重复
以统一社会信用代码为主键，辅以名称相似度比对
Step 2选择"主记录"
选择交易量最大/数据最完整的记录作为主BP
Step 3数据合并
将其他记录的有效数据（联系人/地址）合并到主BP
Step 4历史引用重定向
将已停用的重复BP编码映射到新BP（保留历史查询能力）
Step 5MDG-BP激活
在MDG-BP中创建合并后的标准BP记录

📝 知识测验

Q1. 华灯集团将35家关联方（既是客户又是供应商）统一为双重身份BP，其核心优势是什么？
✅ 解析：双重身份BP的核心价值：同一法律实体只有一个BP编号，统一维护基本信息（名称/地址/银行账号），通过Role分别管理其作为客户和供应商的业务数据，消除信息冗余和不一致。
Q2. HLD_BP_BANK_CHG（银行账号变更CR）为什么需要两步审批（MM Owner + 财务Controller）？
✅ 解析：双人审批是防欺诈的关键内控：内部员工（甚至外部黑客）若能独自完成银行账号变更，可以将付款引导到自己账户。双人审批确保需要两个独立人员协同才能完成变更。
Q3. 重复客户合并时，应该以哪个字段作为主要识别键？
✅ 解析：统一社会信用代码是中国法人实体的唯一国家标识，是识别重复客户的最可靠依据。名称可能有多种写法，电话可能变更，信用代码是不变的唯一ID。
Q4. FLCU01这个BP Role代表什么？
✅ 解析：FLCU01 = Customer（Company Code Data）——客户在特定公司代码下的账务数据，包含应收账款科目、付款条款、信用控制区等。对应ECC的KNB1表。
Q5. 为什么Greenfield项目推荐重新规划BP号码范围，而不是沿用ECC的客户/供应商编码？
✅ 解析：Greenfield是编码重置的最佳时机。代价是下游系统需更新引用，但收益是彻底消除ECC历史遗留的编码混乱（如客户号从1起，供应商号也从1起，两者可能冲突）。
