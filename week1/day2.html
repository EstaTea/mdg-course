Day 2 · MDG 架构与部署模式

理解MDG的技术架构层次，以及Hub vs Co-Deploy 如何选择
⏱ 约55分钟📖 架构 + 对比

2.1 MDG 技术架构三层

层次名称作用关键技术
第一层数据模型层（Data Model）定义主数据结构：实体、属性、关系MDG Data Modeler（SE80/自定义）
第二层应用层（Application Framework）管理变更请求（CR）全生命周期：暂存、校验、审批、激活Change Request Framework (CRF)、BRF+
第三层用户界面层（UI）提供业务用户操作界面SAP Fiori（FioriElements + Freestyle）
架构记忆口诀：数据模型→变更请求→Fiori界面，三层各自独立配置，但相互依赖。

2.2 核心概念：Change Request（变更请求）

MDG的所有主数据操作都以 Change Request（CR） 为单位管理。一个CR包含：

  • CR类型（Change Request Type）：定义这是"新建物料"还是"修改客户地址"等操作类型
  • 暂存数据（Staging Data）：申请的主数据内容，审批前不写入Active
  • 工作流步骤：由CR类型决定审批流是1步还是多步
  • 状态机：草稿→提交→审批中→已激活/已拒绝
重要：一个CR只能对应一个业务对象（如一条物料记录）。需要批量创建100条物料，需要100个CR，或使用MDG的批量处理功能（File Upload）。

2.3 部署模式：Hub vs Co-Deploy

维度Hub部署（独立）Co-Deploy（嵌入S/4HANA）
架构独立SAP系统作为MDG主节点，S/4H通过DRF接收MDG直接安装在S/4HANA实例上
适用场景多ERP系统（SAP+非SAP混合）单一或少量S/4H系统
成本高（需额外服务器许可）低（无需额外系统）
复杂度高（网络、接口、同步机制）低
性能与生产系统隔离共享资源，高峰期可能影响
佛照推荐——✅ 推荐（单一S/4HANA PCE）
佛照决策：佛照本次实施S/4HANA PCE（私有云），只有一套系统，推荐 Co-Deploy 模式，MDG直接运行在S/4HANA上，无需额外系统许可，DRF配置最简单。

MDG 双区隔离设计：Staging Area vs Active Area

📋 Staging Area
• 存储待审批的变更请求数据
• 对业务系统 完全不可见
• SAP 专用暂存表（USMD* 系列）
• 审批拒绝 → 数据丢弃
• 审批通过 → 写入 Active Area

✅ Active Area
• 正式主数据，业务系统可见

2.4 MDG Active Area vs Staging Area

MDG最重要的设计思想——双区分离：

区域含义存储位置谁能写
Staging Area（暂存区）变更请求中的待审批数据，对业务不可见MDG专用staging表（USMD*表）CR发起人 + 审批流程
Active Area（活跃区）审批通过、已激活的正式主数据SAP标准主数据表（MARA/KNA1等）仅MDG激活动作

这个设计确保了：审批通过前，任何变更对生产业务都不可见，彻底杜绝"先进系统再补审批"的情况。

2.5 MDG技术组件全貌

组件用途顾问需要会的操作
MDG Data Modeler定义/扩展数据模型（实体/属性）查看标准模型，了解如何添加自定义字段

MDG EditionMDG-M/BP/G的标准配置包激活正确的业务函数组（NWBC/IMG）

2.6 MDG与S/4HANA的关系

S/4HANA从1709版本起，MDG作为核心组件内置，无需单独安装，只需激活对应的Business Function（业务函数）即可。

激活路径（Co-Deploy）：
SPRO → SAP Reference IMG → Cross-Application Components → Master Data Governance
或：NWBC → MDG Management Console → Activate Business Functions

激活后，Fiori Launchpad上会出现MDG相关的Tile（应用图标），业务用户即可开始使用。

📝 知识测验

Q1. Co-Deploy部署模式与Hub部署模式最核心的区别是什么？
✅ 解析：Co-Deploy：MDG与S/4HANA共用一套系统，适合单一ERP场景，成本低。Hub：MDG独立部署，适合多ERP混合环境。
Q2. MDG的Staging Area（暂存区）存储在哪里？
✅ 解析：MDG有自己的staging表空间（USMD开头），与Active的标准主数据表（MARA/KNA1等）完全隔离。
Q3. Change Request（变更请求）的核心作用是？
✅ 解析：CR是MDG管控主数据变更的核心单元，所有创建/修改/停用操作都必须通过CR，经审批后才能激活。
Q4. BRF+在MDG中主要承担什么职责？
✅ 解析：BRF+（Business Rules Framework plus）在MDG中有两个关键用途：①数据质量校验（必填字段、格式检查）②工作流审批人确定规则（谁来审批这条CR）。
Q5. 佛照项目推荐Co-Deploy而非Hub部署，最主要原因是？
✅ 解析：Hub适合多ERP（SAP+非SAP）混合环境。佛照仅有一套S/4HANA PCE，Co-Deploy成本最低、复杂度最低，是标准推荐方案。
